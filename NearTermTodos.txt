## Near-Term — Comprehensive VM Testing (Critical)

The QEMU VM test suite needs significant enhancement to verify all the new
functionality, especially after the scanner/transparent-compression fixes.
This is the top priority before any release.

- [ ] Add E2E test: transparent compression roundtrip with kprobe
      Enable scanner, allocate anonymous pages, wait for sweep, touch pages,
      verify data integrity after decompression. Currently only the
      compress_vaddr debugfs interface tests PTE replacement — we need a
      real-world test where the scanner compresses and the fault handler
      decompresses.
- [ ] Add E2E test: module unload with active compressed pages
      Compress pages, verify drain_and_restore restores all PTEs,
      check that processes don't SIGBUS after unload.
- [ ] Add E2E test: scanner skip filters
      Verify that VM_LOCKED VMAs, mlocked pages, and shared pages are
      skipped. Check that skip counters (scanner_skip_vma_locked,
      scanner_skip_page_mlocked, scanner_skip_page_shared) increment.
- [ ] Add E2E test: adaptive scan interval
      Verify that interval backs off when no pages are compressed and
      resets when pages are compressed. Check scanner_cycles_total and
      scanner_cycles_empty counters.
- [ ] Add E2E test: incompressible page skip-list
      Verify that recently-incompressible pages are skipped and that
      scanner_skip_incompressible counter increments.
- [ ] Add E2E test: graceful shutdown with compressed pages
      Start with compressed pages, issue `rmmod minimem`, verify that
      drain_and_restore completes and no SIGBUS occurs.
- [ ] Add E2E test: CPU overhead measurement
      Measure minimem_scand CPU usage during active scanning vs idle.
      Verify that adaptive backoff reduces CPU usage when no compression
      is happening. Target: <5% single-core during active scan on 4-CPU VM.
- [ ] Add E2E test: decompression latency measurement
      Measure fault handler latency (time from page fault to present PTE).
      Target: <10μs per 4KB page decompression.
- [ ] Add E2E test: large memory VM (4GB+)
      Test with larger VM to verify scanner doesn't consume excessive CPU
      on systems with many processes and GBs of memory.

## Near-Term — Unified Memory / Heterogeneous Architectures

- [ ] Research: How do unified memory machines (Ryzen AI Max, Strix Halo,
      Apple M-series) partition memory between CPU and GPU blocks?
      Are CPU-compressed pages visible to GPU? Can GPU faults occur if
      the CPU compresses pages that the GPU might need?
- [ ] Research: CXL Type-3 memory and MiniMem. CXL memory IS visible to
      mm/ — our kernel module already works. Verify that scanner correctly
      handles CXL-attached memory pages.
- [ ] Research: For machines with large RAM (128GB+), verify that the
      scanner's adaptive interval, batch throttling, and cond_resched()
      keep CPU usage acceptable. The bloom filter skip-list should help
      with the working set.
- [ ] BPF tracing integration: Add tracepoints for compression start/end,
      decompression start/end, and scanner sweep start/end so that BPF
      tools (bpftrace, bcc) can observe MiniMem's behavior without dmesg.
